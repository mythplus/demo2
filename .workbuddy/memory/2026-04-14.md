### 记忆状态 active 来源确认
- `active/paused/deleted` 不是 Mem0 SDK 原生业务状态，而是本项目自定义的 metadata.state 约定
- 后端在 `server/config.py` 中定义 `VALID_STATES = {"active", "paused", "deleted"}`
- 读取时默认 `metadata.state` 缺失即按 `active` 处理；筛选时也是围绕 Qdrant 的 `metadata.state` 做过滤
- 结论：Mem0 负责存储/透传 metadata，这套 active 功能逻辑是项目侧封装出来的

### 外部仓库 openmemory 状态能力确认
- 检查 `https://github.com/mythplus/mem0` 的 `openmemory` 子项目后确认：openmemory 自身原生实现了记忆状态系统
- `openmemory/api/app/models.py` 定义 `Memory.state = Enum(MemoryState)`，默认 `active`，状态枚举包含 `active/paused/archived/deleted`
- `openmemory/api/app/schemas.py` 的 `MemoryUpdate` 与 `MemoryResponse` 暴露 `state` 字段
- `openmemory/api/app/routers/memories.py` 提供基于状态的读写逻辑：创建为 `active`、暂停为 `paused`、归档为 `archived`、删除为 `deleted`，并记录 `MemoryStatusHistory`
- 结论：如果对比 openmemory，本项目的 active 功能不是照搬 Mem0 SDK，而更接近 openmemory 应用层的状态设计思路

### 记忆状态改造方案建议
- 对比结论：本项目当前是 `metadata.state` 轻量方案，仅有 `active/paused/deleted`；openmemory 是应用层原生状态机方案，含 `archived`、状态历史、归档/删除时间戳与专门状态变更接口
- 建议改造方向：保留 Mem0 + Qdrant 主体不动，在项目层补齐 `archived`、统一 `update_memory_state()` 服务、显式状态历史表、`archived_at/deleted_at/state_updated_at` 字段与恢复/归档接口
- 若要更贴近官方实现，最好补一个 SQLite 状态索引/历史层，避免完全依赖 Qdrant metadata 的隐式默认 `active` 语义
